1. Project Overview

What is OpenViking?

OpenViking is a context database designed for AI agents, using a "filesystem paradigm" to uniformly manage the memory, resources, and skills required by agents.

It addresses core pain points in AI agent development: fragmented context, poor retrieval quality, and lack of observability, enabling developers to manage agent context like local files.

Core Pain Points and Solutions

Pain Point OpenViking Solution
Context Fragmentation Unified filesystem management paradigm
Context Demand Surge L0/L1/L2 three-layer context loading mechanism
Poor Retrieval Quality Recursive directory retrieval strategy
Unobservable Visualized retrieval trails
Non-iterative Memory Automatic session management and memory extraction

4. Business Layer (Detailed)

Business Layer Overview

The Business Layer in OpenViking is the core service layer located in openviking/service/. It provides high-level business logic and operations that abstract the underlying storage and filesystem layers.

The OpenVikingService class (openviking/service/core.py) acts as the central orchestrator, composing all sub-services and managing the infrastructure lifecycle.

4.1 OpenVikingService - Central Orchestrator

Core Service OpenVikingService

Main service class that composes all sub-services and manages infrastructure lifecycle.

File: openviking/service/core.py
Key Responsibilities:
  • Service Composition - Composes all sub-services (FS, Search, Session, etc.)
  • Infrastructure Management - Initializes and manages storage resources, AGFS client, VikingDB, QueueManager
  • Directory Initialization - Creates preset directory structure for accounts, users, and agents
  • Processor Initialization - Initializes ResourceProcessor, SkillProcessor, SessionCompressor
  • Security - Bootstrap encryption module and lock management
Key Properties:
  • fs - FSService instance
  • relations - RelationService instance
  • pack - PackService instance
  • search - SearchService instance
  • resources - ResourceService instance
  • sessions - SessionService instance
  • debug - DebugService instance

6. Deployment Architecture (Docker)

Docker Compose Configuration

OpenViking uses Docker Compose for containerized deployment with the following configuration:

version: "3.8"

services:
  openviking:
    image: ghcr.io/volcengine/openviking:latest
    container_name: openviking
    ports:
      - "1933:1933"  # HTTP API
      - "8020:8020"  # Other services
    volumes:
      - /var/lib/openviking/ov.conf:/app/ov.conf
      - /var/lib/openviking/data:/app/data
    healthcheck:
      test: ["CMD-SHELL", "curl -fsS http://127.0.0.1:1933/health || exit 1"]
      interval: 30s
      timeout: 5s

      start_period: 30s

7.3 Retrieval Flow

Retrieval Strategy - Five Steps:
  1. Intent Analysis → Generate multiple retrieval conditions
  2. Initial Positioning → Vector retrieval quickly locates high-score directories
  3. Refinement & Exploration → Perform secondary retrieval within that directory
  4. Recursive Drill-down → If subdirectories exist, recursively perform secondary retrieval
  5. Result Aggregation → Retrieve the most relevant context and return

9. RAGFS Detailed Documentation

What is RAGFS?

RAGFS = Retrieval-Augmented Generation FS (File System)

RAGFS is OpenViking's core filesystem abstraction layer, an aggregated filesystem implemented in Rust, providing a unified file operation interface for AI agents, allowing agents to operate distributed context data like local files.

RAGFS Core Uses

RAGFS Key Advantages

  • Unified Interface: Same FileSystem trait for different backends
  • Plugin-based: Flexibly add new storage backends
  • High Performance: Rust implementation with async I/O
  • Extensible: Support mounting multiple path types
  • Unified Protocol: Unified resource locator via viking:// protocol
